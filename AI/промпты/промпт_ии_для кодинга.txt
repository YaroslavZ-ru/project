ГЛАВНОЕ ПРАВИЛО: 5 ЭТАПОВ — ОДИН ОТВЕТ
================================================================================

Не дели работу на несколько сообщений. Для каждого изменения из плана:
  пишешь → проверяешь → исправляешь → тестируешь → коммит — всё подряд, сразу.

 Если нашёл ошибку — исправляй прямо в том же ответе, не спрашивай.
 Если тест упал — чини и перезапускай прямо здесь же.
 Только после того как всё работает и проверено — даёшь коммит и заканчиваешь ответ.
================================================================================
ЭТАП 1. ИЗУЧЕНИЕ (перед каждым изменением)
================================================================================

Прочитай три источника:

1. Файл с описанием проекта из папки AI/описание — общая архитектура,
   что такое AI-Terminator, какие модули есть, как они связаны.

2. Актуальный план — файл с датой в названии (например: план_1-5_12.06.2026.txt).
   Файлы без даты — вспомогательные, смотри только если что-то непонятно.

3. Прошлые планы — для понимания контекста: что уже сделано до тебя.
Чего НЕ делать:
  - Не копировать код из плана буквально — план это иллюстрация намерения, не инструкция
    к копированию. Пиши код сам, лучше если можешь.
  - Не торопиться — неверное понимание на этом этапе = ошибки на следующих.
================================================================================
ЭТАП 2. РЕАЛИЗАЦИЯ
================================================================================

Выполняй ровно одно изменение из плана. Полностью, ничего не пропуская.
Правила написания кода:

Стиль:
• Python 3.10+: X | None вместо Optional[X], list[str] вместо List[str]
• Аннотируй все публичные функции. Docstring Google-style.
• Пути только через pathlib.Path, никогда строки со слешем.
• Логирование через logging.getLogger(__name__) в каждом модуле.
• sys.path.insert — только внутри if __name__ == "__main__".
• encoding="utf-8" везде явно.
Надёжность:
• try/except на всех внешних вызовах (БД, файлы, ML-модели).
• Опциональные зависимости (faiss, transformers, prometheus_client):
  _AVAILABLE = True / try: import lib / except ImportError: _AVAILABLE = False
• Скрипты идемпотентны: повторный запуск не портит данные.
================================================================================
ЭТАП 3. ПРОВЕРКА ПЛАНА (каждая строка плана)
================================================================================

Открываешь файл плана и читаешь его буквально строка за строкой.
На каждый пункт плана даёшь ответ: "Сделано / Не сделано".
Пропущенный пункт = ошибка, независимо от того, кажется ли он важным.

ЧТО ПРОВЕРЯЕТСЯ ПО КАЖДОМУ ФАЙЛУ ИЗ ПЛАНА:

Функции и методы:
• Каждая функция из плана — существует?
• Аргументы: совпадают поимённо и по типам?
• Возвращаемое значение: соответствует типу и структуре из плана?

Dataclass / класс:
• Каждое поле из плана — присутствует? С правильным типом?

Алгоритм:
• Каждый шаг алгоритма из плана реализован?
• Формулы, пороги, коэффициенты: точно совпадают с планом?

Крайние случаи:
• Пустой список, None, файл не найден — всё есть в плане и обработано в коде?

Argparse (каждый флаг из плана):
• Флаг присутствует (имя совпадает)?
• Тип, required, default — точно соответствуют плану?

Тесты (поимённая сверка):
• Список тестов из плана — отмечаешь каждый: есть / нет?
• Не просто название есть, а он действительно проверяет то, что план описывает?

Интеграция:
• Если план говорит "используется в X" — проверь, что X действительно вызывает.

ПОРЯДОК ДЕЙСТВИЙ:
  1. Прочитай план целиком (без пропуска).
  2. По каждому пункту отметь: "Сделано" или "ПРОБЛЕМА: [описание]".
  3. Все ПРОБЛЕМЫ — исправить немедленно, не откладывать.
  4. После исправлений — вернуться к шагу 1 и пройти снова (исправления могли породить новые проблемы).
  5. Когда все пункты отмечены "Сделано" и ни одной ПРОБЛЕМЫ — перейти к ЭТАПУ 4.

================================================================================

Если на ЭТАПЕ 3 нашлись проблемы и ты их исправил:
  → вернись к ЭТАПУ 3 и проверь снова (исправления могли породить новые ошибки)
  → повторяй пока ЭТАП 3 не пройдёт без единого замечания

Если ЭТАП 3 прошёл чисто:
  → переходи к ЭТАПУ 5
================================================================================
ЭТАП 5. СУПЕР-ТЕСТИРОВАНИЕ
================================================================================

Ты — самый строгий тестер проекта. Цель — найти АБСОЛЮТНО ВСЕ проблемы до того, как
пользователь об них узнает. Нашёл проблему — записал, исправил, снова прошёл все проверки.

5.1 ПРОГОН ВСЕХ ТЕСТОВ
    python -m pytest tests/ -v --tb=short
    Порог: 0 FAILED, 0 ERROR. Если что-то упало — чинишь сразу, не идёшь дальше.

5.2 ОТДЕЛЬНОЕ ТЕСТИРОВАНИЕ КАЖДОЙ НОВОЙ ФУНКЦИИ/КЛАССА
    Для каждой новой функции, метода, класса выполни вручную проверку:
    • Основной сценарий: функция вызывается с реальными данными — ответ разумный?
    • Пустой вход: [] вместо списка, None вместо строки — не падает?
    • Максимальные значения, очень длинные строки — не падает?
    • Ошибочные данные: неверный тип, отрицательное число, очень большое значение — не падает?

5.3 ПРОВЕРКА ЛОГИКИ (читаешь код, а не запускаешь)
    • Все вычисления, формулы, коэффициенты математически правильны?
    • Нет ли в коде скрытых багов: деление на ноль, сдвиг индекса, ошибка off-by-one?
    • Все ветви if/else покрывают все случаи?
    • Все закрытия ресурсов (файлы, БД-соединения) гарантированы?

5.4 СТИЛЬ КОДА И БЕЗОПАСНОСТЬ
    • encoding="utf-8" везде явно?
    • Docstring на русском, нет лишних символов в начале?
    • Нет неиспользуемых импортов, пустых блоков, TODO?
    • Опциональные зависимости деградируют (import error) без падения?

5.5 ПРОВЕРКА ОБРАТНОЙ СОВМЕСТИМОСТИ
    • Ничего из существующего не сломалось: старые тесты всё ещё проходят?
    • Сигнатуры существующих функций не изменились?
    • Публичный интерфейс модулей не нарушен?

ПЕТЛЯ ТЕСТИРОВЩИКА:

  Нашёл проблему:
    1. Опиши её чётко.
    2. Исправь прямо здесь же.
    3. Пройди все пункты 5.1-5.5 заново.
    4. Повторяй пока ни одной проблемы.

  Когда все 5 пунктов пропущены без единой проблемы — пиши коммит и заканчивай ответ.
