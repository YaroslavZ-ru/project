================================================================================
ШАГ 1. ИЗУЧЕНИЕ ПРОЕКТА (выполняется один раз в начале сессии)
================================================================================

1.1. Прочитай файл описания проекта из папки AI:
     - файл с "финал" или "описание_проекта_финал" в названии
     Это даёт общее понимание: что такое AI-Terminator, зачем он нужен,
     как работает, из каких модулей состоит.

1.2. Найди актуальный план для выполнения.
     Планы лежат в папке AI и имеют дату в названии (например: план_1-5_12.06.2026.txt).
     Выбирай именно файлы с датой — они финальные.
     Файлы без даты или с пометками "уже_сделано", "архив" — вспомогательные,
     используй их только для справки если что-то непонятно.

1.3. Прочитай план внимательно.
     Запомни: код в планах — это ИЛЛЮСТРАЦИИ, а не инструкции для копирования.
     Твоя задача — понять СУТЬ изменения и реализовать его так,
     как ты считаешь правильным и качественным.
     Следуй описанному поведению и структуре, но пиши код сам.

================================================================================
ШАГ 2. ПОДГОТОВКА СРЕДЫ
================================================================================

Считай, что у пользователя установлен только Python (3.10+).
Всё остальное нужно установить.

2.1. Установи зависимости из requirements.txt:
     pip install -r requirements.txt

2.2. Если нужны дополнительные пакеты которых нет в requirements.txt —
     установи их и добавь в requirements.txt с версией.

2.3. Если нужна инициализация базы данных, индексов или других артефактов —
     выполни соответствующие скрипты из папки scripts/.

================================================================================
ШАГ 3. РЕАЛИЗАЦИЯ ИЗМЕНЕНИЙ ИЗ ПЛАНА
================================================================================

3.1. Выполняй изменения по очереди, каждое отдельно и полностью.

3.2. Для каждого изменения:
     - Пойми что нужно сделать (модуль, функция, поведение)
     - Посмотри существующий код в src/ — чтобы не конфликтовать
     - Напиши реализацию: читаемо, с docstring, type hints, обработкой ошибок
     - Следуй принципам проекта: pathlib для путей, логирование через logging,
       импорты вида "from src.module import ..." без sys.path.append

3.3. Стиль кода:
     - Python 3.10+: используй X | Y вместо Union[X, Y], list[str] вместо List[str]
     - PEP 8: snake_case для переменных и функций, CamelCase для классов
     - Все пути через pathlib.Path, никогда не конкатенируй строки с "/" или "\"
     - Логируй всё важное, оборачивай внешние вызовы в try/except

================================================================================
ШАГ 4. ПРОВЕРКА НАПИСАННОГО
================================================================================

4.1. После написания каждого модуля — запусти его и убедись что он работает.
     Не оставляй непроверенный код.

4.2. Запусти тесты если они есть в папке tests/:
     python -m pytest tests/ -v

4.3. Если тестов для нового модуля нет — напиши простую проверку сам
     (в блоке if __name__ == "__main__": или отдельным файлом в tests/).
     Проверь:
     - Основной сценарий работает правильно
     - Краевые случаи не ломают программу (пустые строки, None, отсутствие файла)

4.4. Если что-то упало — исправь, не двигайся дальше пока не заработает.

================================================================================
ШАГ 5. СВЕРКА С ПЛАНОМ
================================================================================

После выполнения всех изменений — сравни что сделано с тем что было в плане.

5.1. Пройдись по каждому пункту плана и проверь: выполнен ли он.

5.2. Если что-то пропущено или сделано частично — доделай.

5.3. Если в плане был пункт который уже был реализован ранее —
     убедись что существующая реализация соответствует описанию, при необходимости доработай.

================================================================================
ШАГ 6. ОФОРМЛЕНИЕ КОММИТА
================================================================================

После того как всё готово, проверено и работает — оформи коммит.

6.1. НАЗВАНИЕ коммита (до 50 символов, на русском):
     - Кратко и по делу: что именно добавлено или изменено
     - Начинай с глагола в прошедшем времени: "Добавил", "Реализовал", "Исправил"
     - Без технического жаргона и аббревиатур без расшифровки
     Пример: "Добавил модуль лемматизации с кешированием"

6.2. ОПИСАНИЕ коммита (3-8 строк, на понятном языке):
     - Что теперь умеет система, чего не умела раньше
     - Какие файлы появились или изменились
     - Что теперь работает и как это можно проверить
     - Никакого технического жаргона — пиши как для человека который не программист

     СТРОГО ЗАПРЕЩЕНО в коммите:
     - Упоминать номера изменений: "Изменение 3", "изм. 5" и т.п.
     - Ссылаться на план: "согласно плану", "по плану", "план 1-5" и т.п.
     - Упоминать будущие шаги: "будет добавлено позже", "подключится в изменении 7" и т.п.
     - Ставить кавычки вокруг имён файлов, функций и значений — пиши их без кавычек
     Коммит описывает только то что реально сделано прямо сейчас.

     Пример:
       Теперь система умеет приводить слова к их начальной форме (бежал -> бежать).
       Добавлен файл src/lemmatizer.py.
       Результаты сохраняются в кеш — повторные запросы работают мгновенно.
       Проверить можно запустив: python -m pytest tests/test_lemmatizer.py -v

6.3. Выведи результат в таком формате:

     --- КОММИТ ---
     Название: <название>

     Описание:
     <описание>
     ---------------
