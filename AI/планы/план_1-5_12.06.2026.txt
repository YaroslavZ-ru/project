ОБЩИЕ ПРИНЦИПЫ (применяются ко всем изменениям этого и последующих планов)
--------------------------------------------------------------------------------

Эти принципы — фундамент всего проекта. Прочитайте их один раз внимательно.

1. ЯЗЫК И ВЕРСИЯ PYTHON
   Весь код пишется на Python 3.10 или выше. Используйте современный синтаксис:
   объединение типов через | вместо Union, встроенные дженерики list[str] вместо
   List[str] и т.п. Проект НЕ должен зависеть от Python 3.8 или 3.9.

2. СТРУКТУРА ПРОЕКТА КАК ПАКЕТА
   Корневая папка проекта (где лежит main.py) — точка отсчёта. Все модули находятся
   в подпапке src/. src/ является пакетом (есть __init__.py), что позволяет делать
   импорты вида 'from src.config import Config'. Никаких sys.path.append.

3. КРОССПЛАТФОРМЕННОСТЬ ПУТЕЙ
   Все пути строятся ТОЛЬКО через pathlib.Path. Никогда не конкатенируйте строки
   с '/' или '\'. Это одинаково работает на Windows, Linux, macOS.

4. ПРОВЕРКА РАБОТОСПОСОБНОСТИ ПОСЛЕ КАЖДОГО ИЗМЕНЕНИЯ
   Каждое изменение завершается содержательным тестом: папка создалась, конфиг
   загрузился без ошибок, заглушка вернула JSON нужной структуры. Результат
   проверки должен быть виден в консоли или в лог-файле.

5. ПРИМЕРЫ КОДА — ТОЛЬКО ИЛЛЮСТРАЦИИ
   Все фрагменты кода в этом плане показывают НАМЕРЕНИЕ, а не точную реализацию.
   Адаптируйте их под свой стиль и соглашения проекта.

6. ОБРАБОТКА ОШИБОК ВЕЗДЕ
   Любой внешний вызов (открытие файла, импорт модели, запрос к БД) должен быть
   обёрнут в обработку исключений. Пользователь получает понятное сообщение,
   а не технический stack trace.

7. ДОКУМЕНТИРОВАНИЕ ОБЯЗАТЕЛЬНО
   Каждый модуль, класс и публичная функция — docstring в стиле Google
   (секции Args:, Returns:, Raises:). Комментарии к нетривиальным участкам кода.

8. ТИПИЗАЦИЯ ВЕЗДЕ
   Type hints для всех параметров и возвращаемых значений. Это облегчает понимание
   кода и позволяет IDE и mypy находить ошибки заранее.

ДЕТАЛЬНОЕ ОПИСАНИЕ КАЖДОЙ ПАПКИ

src/ — ядро проекта. Здесь будут создаваться все Python-модули по мере выполнения

работать не будут.

data/ — директория для хранимых данных. Здесь впоследствии окажутся:
- knowledge_base.db (файл SQLite базы знаний, создаётся в изм.11)
- synonyms.json (словарь синонимов, создаётся в изм.7)
На данном этапе папка пустая.

models/ — папка для больших бинарных файлов моделей. Здесь будет лежать
файл cc.ru.300.bin (~8 ГБ) — предобученная fastText-модель. Скачивается
отдельно (подробно — в изменении 11). Сейчас папка пустая.

tests/ — тесты на базе pytest. Уже сейчас создать:
- __init__.py (пустой файл, делает tests пакетом)
- test_config.py (заглушка: может содержать просто комментарий

logs/ — папка для лог-файлов. Программа будет писать сюда файл ai_terminator.log.
ВАЖНО: код должен создавать эту папку автоматически при старте, если
её нет (mkdir parents=True, exist_ok=True).

configs/ — папка для конфигурационных файлов. Здесь будут лежать:
- config.json (изм.4)
- domain_templates.json (изм.14)
- domain_keywords.json (изм.14)

scripts/ — обязательная часть проекта. Без неё БД никогда не будет инициализирована.
Будущие скрипты в этой папке:
- init_db.py — создаёт схему таблиц в SQLite (изм.11)
- seed_db.py — загружает начальные данные (изм.11)
- update_kb.py — импорт новых понятий (позже)

setup_project.py — скрипт, создающий всю структуру одной командой
'python setup_project.py'. Запускается один раз при развёртывании проекта
на новой машине. Делает:
- создаёт все нужные папки (метод mkdir с exist_ok=True),
- создаёт пустые __init__.py в src/ и tests/,
- создаёт заглушку tests/test_config.py,
- проверяет, что main.py существует (но не перезаписывает его),
- выводит в консоль отчёт о выполненных действиях.

выход, но внутри run_pipeline — фиктивные данные. Полноценный пайплайн

ПОЧЕМУ ЗАГЛУШКА, А НЕ ПОЛНОЦЕННЫЙ ПАЙПЛАЙН
-----------------------------------------------
ИМПОРТАНТНО: на данном этапе ЕЩЁ НЕ существуют модули:

  src/vectorize.py   (будет в изм.11-12)

Нельзя их импортировать в main.py сейчас — возникнет ImportError.
Правильный подход: main.py содержит СТРУКТУРУ пайплайна сейчас,
а реальные вызовы модулей будут добавляться постепенно.

ФОРМАТ ВХОДНЫХ ДАННЫХ
---------------------
Входной JSON-объект содержит поля:
  'term' (строка, обязательное) — анализируемый термин, например 'ключ'
  'hints' (список, опционально) — уточняющие слова. Если нет — []
  'debug' (булево, опционально) — если true, добавить debug_info в ответ. Если нет — false
  'min_confidence' (число, опционально) — порог поиска. Если нет — из конфига (0.3)

Пример входного JSON:
  {"term": "ключ", "hints": ["техника", "вращение"], "debug": false}

main.py принимает вход двумя способами (любым из двух, или обоими):
  1. Из stdin: echo '{"term":"ключ"}' | python main.py
  2. Через аргумент: python main.py --input '{"term":"ключ"}'

ФОРМАТ ВЫХОДНЫХ ДАННЫХ
-----------------------
main.py выводит в stdout ТОЛЬКО JSON (ничего лишнего). Логи — в файл
или в stderr.

Структура успешного ответа:
  {
    "status": "ok",
    "term": "ключ",
    "selected_context": {"domain": "не определено", "confidence": 0.0},
    "parameters": [],
    "suggested_refinements": [],
    "warnings": []
  }

Структура ответа при ошибке:
  {"status": "error", "message": "Текст ошибки"}

СТРУКТУРА main.py
------------------
1. ИМПОРТЫ И ЛОГИРОВАНИЕ
   Импорт стандартных библиотек (json, sys, argparse, logging, pathlib).
   Настройка logging: два handler-а:
     FileHandler — в logs/ai_terminator.log (уровень DEBUG, всё детально)
     StreamHandler — в stderr (уровень WARNING, только важное)
   Папка logs/ создаётся автоматически, если отсутствует.

2. ОПРЕДЕЛЕНИЕ КОРНЯ ПРОЕКТА
   PROJECT_ROOT = Path(__file__).parent
   Это значение передаётся в Config (изм.5) и другие модули.

3. ФУНКЦИЯ parse_input(raw: str) -> dict
   Делает:
   - Парсит JSON через json.loads. Если невалиден — ValueError.
   - Проверяет наличие 'term'. Если отсутствует или пусто после strip() — ValueError.
   - Если 'hints' отсутствует ИЛИ равен null — подставляет [].
     Важно: json.loads вернёт None для "hints": null, это не то же самое что
     отсутствие ключа — оба случая нужно явно обработать.
   - Если 'debug' отсутствует — False.
   - Если 'min_confidence' отсутствует — None (будет взято из конфига).
   - Возвращает нормализованный словарь.

4. ФУНКЦИЯ run_pipeline(term, hints, debug, min_confidence, cfg) -> dict
   Центральная функция. Сейчас возвращает фиктивный ответ.
   ВНИМАНИЕ: сигнатура будет расширена при добавлении реальных компонентов.
   В изменении 10 сюда добавятся параметры lemmatizer и synonym_dict.
   Заглушку писать с текущей сигнатурой — это нормально.
   Добавьте комментарий внутри функции, показывающий, что сюда войдут вызовы:
     # Шаг 1: preprocess() из src/preprocess.py (изм.9-10)
     # Шаг 2: vectorize() из src/vectorize.py (изм.11-12)
     # Шаг 3: search_similar_concepts() из src/search.py (изм.17)
     # Шаг 4: aggregate_parameters() из src/aggregation.py (изм.18)
     # Шаг 4-бис: generative_suggest() из src/generative.py (изм.21+)
     # Шаг 5: build_response() из src/response_builder.py (изм.19-20)

5. БЛОК if __name__ == "__main__":
   - Загрузка конфига (Config.from_json - после реализации изм.5;
     пока временно можно использовать None или заглушку словаря).
   - Чтение входа: сначала проверить --input, затем stdin.
   - Вызов parse_input, затем run_pipeline.
   - json.dumps(result, ensure_ascii=False, indent=2) в stdout.
   - Обработка всех исключений: '{"status": "error", "message": "..."}'.
     Завершение с sys.exit(1) в случае ошибки.

ПРОВЕРКА
--------
1. Запустить main.py с тестовым входом. В stdout — валидный JSON
   с полями: status, term, selected_context, parameters,
   suggested_refinements, warnings.
2. В папке logs/ появился ai_terminator.log.
3. При невалидном JSON — {'status': 'error', 'message': '...'}.
4. При отсутствии 'term' — та же структура ошибки.

ПОЯСНЕНИЕ КАЖДОГО ПОЛЯ
--------------------
Пути к файлам (все относительные, преобразуются в абсолютные при загрузке через Config):
  db_path — SQLite файл. На этапе может не существовать.
  fasttext_model_path — ~8 ГБ файл. На этапе может не существовать.
  fallback_embeddings_path — путь к .npy-файлу со статическими эмбеддингами
    (резервный режим если fastText не загружен). Пустая строка = fallback отсутствует.
    Нужен начиная с изм.11. На этапе 1-5 оставить пустым.
  synonyms_path — JSON-словарь: 'слово': ['синоним1', '...']. Создаётся в изм.7.
  domain_templates_path — шаблоны параметров для каждого домена.
    Используется в резервном режиме (если понятие не найдено в БД). Создаётся в изм.14.
  domain_keywords_path — словарь ключевых слов для определения домена.
    Создаётся в изм.14.

Параметры поиска:
  min_confidence — минимальное косинусное сходство (0.0-1.0). 0.3 = не менее 30%.
    Если кандидатов мало — порог автоснижается до 0.2.
  max_candidates — макс понятий-кандидатов из БД. Больше = точнее, но медленнее.
  max_parameters — макс параметров в итоговом ответе.

Генеративный модуль (только если use_generative=true):
  use_generative — флаг включения генеративного дополнения. Сейчас false.
  generative_model — полное имя модели (rugpt3small от Сбера, ~500 МБ).
  generative_max_new_tokens — макс новых токенов за один вызов.
  generative_temperature — температура генерации (0.7 = баланс).
  generative_max_new_params — макс новых параметров из генеративного модуля.
  generative_timeout_seconds — макс время ожидания ответа модели.
  min_parameters_for_generative — если из БД > этого числа параметров —
    генеративный модуль не вызывается.
  generative_keywords — ключевые слова для эвристики типа параметра.

Технические параметры:
  timeout_seconds — общий таймаут операции поиска (защита от зависания).
  cache_embeddings — кэшировать эмбеддинги в LRU-кэше.
  log_level — DEBUG/INFO/WARNING/ERROR.
  cache_lemma_size — размер LRU-кэша лемматизации.
  max_synonyms_per_token — макс синонимов для одного токена (шаг 1 конвейера).
  use_synonyms — если false — синонимы не добавляются вовсе.
  max_term_length — макс длина термина в символах после очистки.
  max_hint_length — макс длина одной подсказки после очистки.
  word_vector_cache_size — размер кэша векторов отдельных слов.
  query_cache_size — размер кэша векторов запросов.

FAISS параметры:
  use_faiss — включить FAISS-индекс (только если БД > 10 000 понятий). Пока false.
  faiss_index_path — путь к индексу. Пустая строка = не используется.

Параметры сессий:
  session_ttl_seconds — время жизни сессии (сек). 1800 = 30 мин.
  session_cache_size — макс одновременных сессий в памяти.
  session_cleanup_interval_seconds — частота очистки устаревших сессий.
  auto_save_domain_on_ok — запоминать домен в сессии при успехе.

Омонимия и домены:
  ambiguity_threshold — порог определения неоднозначности термина.
  ambiguity_delta — допустимая разница между топ-1 и топ-2 кандидатами.
  domain_centroid_threshold — мин сходство с центроидом домена.
  auto_save_domain_on_fallback — запоминать домен если ответ из шаблона (обычно false).

Онтология:
  use_relations — учёт связей is_a, part_of. Пока false.
  relation_max_depth — глубина прохода по иерархии.
  relation_decay_factor — затухание веса при движении вверх по иерархии.
  domain_centroids_min_concepts — мин понятий в домене для центроида.

ИЗМЕНЕНИЕ 5: Класс Config для загрузки и валидации конфигурации

ЧТО НУЖНО СДЕЛАТЬ
-----------------
Создать файл src/config.py с классом Config. Это единственный
способ доступа к конфигурации во всём проекте. Все модули получают
настройки через экземпляр Config.

НАЗНАЧЕНИЕ КЛАССА
-----------------
Config делает четыре вещи:
1. Загружает данные из configs/config.json.
2. Преобразует строковые относительные пути в объекты pathlib.Path с
   абсолютными путями.
3. Валидирует типы и допустимые значения каждого поля.
4. Предоставляет доступ к параметрам как к атрибутам (config.db_path, config.min_confidence).

ОПРЕДЕЛЕНИЕ КОРНЯ ПРОЕКТА
-----------------------
Ключевой вопрос. Config должен знать, где находится корень проекта,
чтобы превращать 'data/knowledge_base.db' в абсолютный путь.

Рекомендуемый способ: передавать project_root явно в метод
фабрики from_json. В main.py это будет:
  PROJECT_ROOT = Path(__file__).parent
  cfg = Config.from_json('configs/config.json', project_root=PROJECT_ROOT)

Альтернатива: автоматически искать родительскую папку,
содержащую подпапку 'configs'. Но это менее надёжно.

СТРУКТУРА КЛАССА (рекомендуется @dataclass)
------------------------------------
Использовать @dataclass. Каждый ключ из config.json — поле класса.
Поля без дефолтов (обязательные) — до полей с дефолтами (опциональные).

Обязательные пути (тип Path после преобразования):

  domain_templates_path, domain_keywords_path

Опциональный путь (преобразуется только если непустой):
  fallback_embeddings_path: str = ''
  (пустая строка = нет fallback; непустая — преобразуется в абсолютный Path.
   Не добавлять в список _path-полей для автоматической конвертации,
   обрабатывать отдельно: if value: data[key] = (project_root / value).resolve())

Числовые параметры поиска (обязательные):

МЕТОД from_json (classmethod)
-----------------------------
Фабричный метод. Принимает: путь к JSON и project_root (Path).

Порядок действий:
1. Проверить наличие config.json. Если нет — FileNotFoundError с
   сообщением 'Файл конфигурации не найден: <путь>'.

2. Открыть с encoding='utf-8', прочитать через json.load. Если
   JSON невалиден — ValueError 'Ошибка парсинга config.json: <подробности>'.

3. Преобразовать пути. Все поля, имя которых заканчивается на '_path'
   (кроме faiss_index_path), преобразуются:
   - если строка непустая: data[key] = (project_root / data[key]).resolve()
   - faiss_index_path: если пустая строка — оставить пустой строкой ''
     (тип str, не None — это согласуется с объявлением в dataclass: str = '').
   - Для db_path: проверить, что родительская директория существует.
     Если нет — выдать предупреждение (logger.warning), но НЕ создавать
     автоматически из Config. Создание директории data/ — задача
     setup_project.py и init_db.py, а не класса конфигурации.

4. Валидация (ValueError если неверно, с понятным сообщением):

   min_confidence: float в [0.0, 1.0] включительно.
     'ошибка: min_confidence должен быть от 0 до 1, получено: <значение>'

   max_candidates, max_parameters, max_term_length, max_hint_length,
   max_synonyms_per_token, word_vector_cache_size, query_cache_size,
   session_ttl_seconds, session_cache_size, session_cleanup_interval_seconds,
   cache_lemma_size, relation_max_depth, domain_centroids_min_concepts,
   generative_max_new_tokens, generative_max_new_params,
   min_parameters_for_generative:
     все должны быть целыми положительными (int > 0).

   timeout_seconds, generative_timeout_seconds:
     float > 0. 'таймаут должен быть положительным'

   log_level: строка, одна из ('DEBUG','INFO','WARNING','ERROR').
     'ошибка: log_level должен быть одним из DEBUG/INFO/WARNING/ERROR'

   use_generative: bool. Если true, то generative_model должна
     быть непустой строкой. 'генеративная модель не задана'

   generative_keywords: list[str]. Может быть пустым,
     но каждый элемент — непустая строка.

   generative_temperature: float в (0.0, 2.0].

   ambiguity_threshold, ambiguity_delta, domain_centroid_threshold,
   relation_decay_factor: float в [0.0, 1.0].

   use_synonyms, cache_embeddings, use_faiss, auto_save_domain_on_ok,
   auto_save_domain_on_fallback, use_relations: bool.

5. Создать экземпляр cls(**data) и вернуть.

ФУНКЦИЯ get_config() (опционально, рекомендуется)
------------------------------------------
Глобальная переменная _config = None и функция get_config().
При первом вызове — загружает и кэширует, при последующих
— возвращает кэш. Исключает повторное чтение файла.

ПРОВЕРКА
--------
1. Загрузить реальный configs/config.json без ошибок.
   cfg.db_path.is_absolute() == True.

2. При min_confidence = 1.5 в JSON — ValueError.

3. При отсутствии max_candidates в JSON — KeyError или ValueError.

4. При log_level = 'info' (строчными) — ValueError с понятным сообщением.

5. После загрузки в main.py добавить запись в лог:
   logger.info('Конфигурация загружена: %s', cfg.log_level)
