ФОРМУЛА ВЕСОВ (из ТЗ, ключевой ориентир для всех изменений)
-------------------------------------------
V = 0.7 * V_term + sum(0.3/N * V_hint_i) + sum(0.1/M * V_syn_j)
где:
  V_term   — взвешенное среднее векторов слов термина
  V_hint_i — вектор i-й подсказки, N — количество подсказок
  V_syn_j  — вектор j-го синонима, M — общее количество уникальных синонимов

ЧТО ЭТО ОЗНАЧАЕТ НА ПРАКТИКЕ:
  - Суммарный вес всех слов термина = 0.7 (равномерно по словам)
  - Суммарный вес всех слов всех подсказок = 0.3 (равномерно по всем словам)
  - Суммарный вес всех уникальных синонимов = 0.1 (равномерно)
  - Если нет подсказок, вес 0.3 за ними просто не используется
  - Если нет синонимов, вес 0.1 просто не используется

ЗАМЕЧАНИЕ О СУММЕ ВЕСОВ:
  Сумма всех весов в tokens_with_weights может превышать 1.0:
  термин(0.7) + подсказки(0.3) + синонимы(0.1) = 1.1.
  Это нормально и заложено в дизайн: vectorize (шаг 2, изм.11-15)
  применяет L2-нормирование финального вектора.
  НЕ пытаться нормализовать веса в preprocess.py — это задача векторизатора.

ИЗМЕНЕНИЕ 6: Лемматизатор (синглтон) с кэшем LRU
--------------------------------------------------------------------------------

ЧТО НУЖНО СДЕЛАТЬ
-----------------
Создать модуль src/lemmatizer.py с классом Lemmatizer.
Он принимает слово/фразу и возвращает лемму (нормальную форму) каждого
слова. Это фундаментальный камень всей предобработки.

БИБЛИОТЕКА
-----------
Использовать pymorphy3 (НЕ pymorphy2!). В изменении 2 была установлена именно pymorphy3.
API полностью совместим: import pymorphy3; MorphAnalyzer() работает так же.

ПОЧЕМУ СИНГЛТОН
------------------
pymorphy3.MorphAnalyzer() загружает в память словарь русского языка. Это
тяжёлая операция, которая должна выполняться ровно ОДИН раз за всё время
работы программы. Правильный способ — синглтон через __new__,
где при первом создании класс создаёт MorphAnalyzer, а при
последующих Lemmatizer() возвращает тот же самый объект.

ПОЧЕМУ LRU-КЭШ С ОГРАНИЧЕНИЕМ РАЗМЕРА
------------------------------------
Без ограничения кэш растёт бесконечно. Для сервера, обрабатывающего тысячи
запросов в день, это проблема. Размер кэша задаётся параметром
из конфига (cache_lemma_size, по умолчанию 1000).

Реализация LRU через collections.OrderedDict:
  - при добавлении: если len >= cache_size: удалить старейший (last=False)
  - при обращении: move_to_end(word)

МЕТОДЫ КЛАССА
--------------
Lemmatizer должен иметь два публичных метода:

1. lemmatize_word(word: str) -> str
   — обрабатывает ОДНО слово (без пробелов и дефисов)

2. lemmatize_phrase(phrase: str) -> list[str]
   — обрабатывает целую фразу

ПОДРОБНОЕ ОПИСАНИЕ lemmatize_word
-----------------------------------
Вход: одно слово (без пробелов, без дефисов).

Шаг 1. Если word пустое или None — вернуть пустую строку.

Шаг 2. Проверить кэш: если word уже есть в кэше —
   переместить в конец (move_to_end), вернуть значение. Лог DEBUG.

Шаг 3. Кэш-промах: вызвать morph.parse(word).

Шаг 4. Выбор лучшего разбора:
   - Если список разборов не пустой: выбрать тот, у которого max(score),
     и взять best.normal_form.
   - Если список пустой (напр., цифры) — вернуть word.lower().
   - Если возникло исключение — перехватить, word.lower(), лог WARNING.

Шаг 5. Сохранить в кэш с соблюдением лимита:
   - Если len(cache) >= cache_size: удалить cache.popitem(last=False)
   - Добавить: cache[word] = lemma
   - Лог DEBUG: 'Лемматизация word -> lemma'

Шаг 6. Вернуть лемму.

ОСОБЫЕ СЛУЧАИ ДЛЯ lemmatize_word
----------------------------------
- Слово состоит только из цифр (напр., '123') — pymorphy3 вернёт пустой
  список; возвращаем слово как есть.
- Слово содержит латинские буквы — pymorphy3 не лемматизирует их;
  возвращаем word.lower().
- Смешанные слова (латинь-кириллица, 'usb-ключ') — после очистки
  дефис будет заменён пробелом; lemmatize_word получает
  чистые слова ('usb', 'ключ') по отдельности.

ПОДРОБНОЕ ОПИСАНИЕ lemmatize_phrase
--------------------------------------
Вход: строка, может содержать пробелы И дефисы.

Очень важно: перед разбивкой нужно заменить ДЕФИСЫ НА ПРОБЕЛЫ.
Иначе 'ключ-гаечный' после split() останется одним
словом и не разобьётся на 'ключ' и 'гаечный'.

Порядок действий:
  1. Если phrase пустое — вернуть []
  2. Заменить все дефисы на пробелы: phrase.replace('-', ' ')
  3. Разбить по пробелам: phrase.split()
  4. Для каждого непустого слова — вызвать lemmatize_word
  5. Отбросить пустые результаты, вернуть список.

ИНИЦИАЛИЗАЦИЯ КЛАССА
-------------------
Конструктор принимает cache_size: int. Если не передан —
брать из конфига (cache_lemma_size). Но так как класс — синглтон,
конфиг нужно передать при первом создании: Lemmatizer(cache_size=cfg.cache_lemma_size).
При повторных Lemmatizer() кэш не сбрасывается.

ПОЧЕМУ ВЕС ЕСТЬ В ФАЙЛЕ, а НЕ ОДИН СПИСОК
-----------------------------------------
Два разных веса часто путают друг с другом. Поясним:

- Вес в файле (weight) — это вес РЕЛЕВАНТНОСТИ синонима.
  Он используется только для РАНЖИРОВАНИЯ: чтобы выбрать ТОП-2
  наиболее релевантных синонима если в списке их больше двух.
  Пример: ключ -> [инструмент(0.8), отмычка(0.4), приспособление(0.2)].
  При max_synonyms=2 возьмём топ-2 по весу: 'инструмент' и 'отмычка'.

- Вес ДЛЯ ВЕКТОРИЗАЦИИ (0.1/M) — это совсем другой вес,
  он вычисляется алгоритмически в preprocess.py и зависит
  от итогового количества уникальных синонимов M.

Таким образом, два веса не путают:
  - файловый weight   -> порядок выбора при отборе топ-N
  - векторный 0.1/M   -> вес токена при векторизации

КЛЮЧИ ФАЙЛА
-----------
Ключи — ТОЛЬКО леммы (именительный падеж, ед. число).
Значения (word) — тоже леммы, не словоформы. Создатель словаря
должен заранее пользоваться лемматизатором при пополнении.
Вес в диапазоне [0.0, 1.0]. Если вес не указан — по умолчанию 0.5.

КЛАСС SynonymDict
-----------------
Файл: src/synonyms.py

Инициализация:
  Принимает путь json_path (str или Path) из конфига (cfg.synonyms_path).
  Внутри класса привести к str: str(json_path).
  Загружает файл. Обрабатывает ошибки:
    - FileNotFoundError: WARNING + self._data = {}
    - json.JSONDecodeError: ERROR + self._data = {}
    - Любое другое искл.: ERROR + self._data = {}
  Успешная загрузка: INFO 'Загружено N словарных статей из <путь>'.

  Дополнительная валидация структуры файла:
    - Корень — dict. Значение каждого ключа — list.
    - Каждый элемент списка — dict с полями 'word' (str) и 'weight' (float).
    - Если 'weight' отсутствует — подставить 0.5.
    - Если 'word' отсутствует или пустой: WARNING, пропустить элемент.

Метод get_synonyms(lemma: str, max_synonyms: int = 2) -> list[str]:
  Если лемма не в словаре — вернуть [].
  Сортировать синонимы по убыванию weight.
  Взять первые max_synonyms штук.
  Вернуть список слов (str), НЕ кортежей.

Опционально: has_synonyms(lemma: str) -> bool.

ПОЧЕМУ ТОЛЬКО СЛОВА
------------------
get_synonyms возвращает list[str], не list[tuple]. Веса
релевантности используются ТОЛЬКО внутри класса для
сортировки. Возвращать кортежи наружу не нужно: preprocess.py
получает чистые слова и сам считает вес (0.1/M).

ПОДКЛЮЧЕНИЕ К main.py
------------------
Создать SynonymDict при старте приложения:

Передавать в run_pipeline как аргумент.
Файл может отсутствовать на начальных этапах — система должна
работать без него (пустой словарь + WARNING в лог).

ЧТО НУЖНО СДЕЛАТЬ
-----------------
Создать модуль src/text_cleaner.py с функцией clean_text(text: str) -> str.
Она принимает сырую строку (термин или подсказка) и возвращает
очищенную строку, готовую к лемматизации.

ЧТО ДОЛЖНО ОСТАТЬСЯ В СТРОКЕ
--------------------------
  - русские буквы (а-я + ё)
  - латинские буквы (a-z)
  - цифры (0-9)
  - пробелы
  - ВНУТРЕННИЕ дефисы (напр., ключ-гаечный)

ЧТО УДАЛЯЕТСЯ
--------------
  - все символы кроме указанных (знаки препинания, спецсимволы, скобки)
  - КРАЕВЫЕ дефисы (где нет буквы слева/справа)
  - дефисы, окружённые пробелами (одиночные дефисы-разделители)
  - лишние пробелы
  - регистр: всё становится нижним

ПОЧЕМУ ВНУТРЕННИЕ ДЕФИСЫ ОСТАЮТСЯ
-------------------------------
Составные слова типа 'ключ-гаечный' или 'USB-ключ' — значимая информация.
Дефис внутри слова сохраняется, чтобы:
  1. Пользователь видел в clean_term читаемое 'ключ-гаечный'
  2. Лемматизатор в lemmatize_phrase сам заменит дефис пробелом

АЛГОРИТМ (7 шагов)
-----------
Шаг 1. Если text не str или None — вернуть ''.

Шаг 2. Привести к нижнему регистру: text = text.lower()

Шаг 3. Заменить символы из неразрешённого набора НА ПРОБЕЛ
(не удалять — иначе 'ключ!гаечный' слипнется в 'ключгаечный'):
  Паттерн: все символы кроме [а-яёa-z0-9 пробел дефис]
  заменять пробелом. Флаг IGNORECASE не нужен (регистр уже нижний).

Шаг 4. Схлопнуть множественные пробелы в один.

Шаг 5. Удалить пробелы и дефисы с краёв строки.

Шаг 6. Убрать дефисы, которые оказались БЕЗ буквы с одной из сторон.
  Этот шаг ОБЯЗАТЕЛЕН. Без него строки типа 'ключ - гаечный' или '-3-болт'
  дадут одиночный дефис в середине результата, что сломает лемматизацию.
  Паттерн удаляет дефис, если слева нет символа не-пробела ИЛИ справа нет:
    text = re.sub(pattern_isolated_hyphen, ' ', text)
  затем снова схлопнуть пробелы и обрезать края.
  Рекомендуемый паттерн: r'(?<!\S)-|-(?!\S)'
  (ищет дефис, у которого слева или справа пробел/начало/конец строки)

Шаг 7. Вернуть результат. Если пустой — вернуть ''.

ПРИМЕРЫ
-------
  '  -Ключ-гаечный! (размер 12)  ' -> 'ключ-гаечный размер 12'
  'USB-ключ'                        -> 'usb-ключ'
  '!!!'                             -> ''
  '-'                               -> ''
  None                              -> ''
  ''                                -> ''
  'Ключ!!!Гаечный'                  -> 'ключ гаечный'
  'ключ - гаечный'                  -> 'ключ гаечный'   (шаг 6)
  '-3-болт'                         -> '3 болт'          (шаг 6)
  'Ключ  --  Гаечный'               -> 'ключ гаечный'

--------
  tests/test_text_cleaner.py. Проверить все примеры выше плюс:
    '  '                -> ''
    'Ключ  --  Гаечный' -> 'ключ гаечный'
    'ключ - гаечный'    -> 'ключ гаечный'  (шаг 6: дефис с пробелами)
    '-3-болт'           -> '3 болт'        (шаг 6: дефис без буквы слева)

НАПОМИНАНИЕ
----------

ИЗМЕНЕНИЕ 9: Базовая предобработка (src/preprocess.py — функция preprocess_base)
--------------------------------------------------------------------------------

ЧТО НУЖНО СДЕЛАТЬ
-----------------
Создать модуль src/preprocess.py. В нём реализовать функцию preprocess_base.
В изм.10 в этом же файле будет добавлена preprocess_full и псевдоним preprocess.
Почему две внутренних функции? preprocess_base можно тестировать изолированно,
без словаря синонимов.

СИГНАТУРА ФУНКЦИИ
------------------
preprocess_base(term: str, hints: list[str] | None, config: Config, lemmatizer: Lemmatizer) -> dict

ВХОДНЫЕ ДАННЫЕ
-------------
  term       — сырая строка термина
  hints      — список сырых подсказок (может быть пустым или None)
  config     — объект Config (доступ к max_term_length, max_hint_length)
  lemmatizer — экземпляр Lemmatizer, созданный в main.py

ИСПОЛЬЗУЕМЫЕ МОДУЛИ
-----------------
  - clean_text из src/text_cleaner.py
  - Lemmatizer из src/lemmatizer.py

ВЫХОДНОЙ СЛОВАРЬ (успех)
-----------------
  {
    'status': 'ok',
    'original_term': str,
    'original_hints': list[str],
    'clean_term': str,
    'clean_hints': list[str],
    'term_lemmas': list[str],
    'hints_lemmas': list[list[str]],
    'all_lemmas': list[str],
    'warnings': list[str]
  }

ПРИ ОШИБКЕ: {'status': 'error', 'message': str}

АЛГОРИТМ
--------
Шаг 1. Нормализация входа:
   Если hints is None — заменить на []. JSON может прислать
   null вместо пустого списка, это допустимо.
   Если term is None — вернуть error 'Термин не передан'.
   Сохранить original_term и original_hints.

Шаг 2. Очистка термина:
   clean_term = clean_text(term)
   Если clean_term == '' — вернуть error 'Термин пуст после очистки'.
   Если len(clean_term) > config.max_term_length — error 'Термин слишком длинный'.

Шаг 3. Очистка подсказок:
   a. Применить clean_text к каждой подсказке.
   b. Отбросить подсказки, ставшие пустыми.
      Если были отброшены — WARNING 'Отброшено N пустых подсказок'.
   c. Проверить длину каждой: > config.max_hint_length — error.
   d. Удалить дубликаты (сохраняя порядок): list(dict.fromkeys(clean_hints))
      Если дубликаты были — WARNING 'Обнаружены дублирующиеся подсказки, они удалены'.

Шаг 4. Лемматизация (lemmatizer передан как аргумент):
   term_lemmas = lemmatizer.lemmatize_phrase(clean_term)
   Если term_lemmas == [] — error 'Термин не содержит значимых слов'.

   hints_lemmas = [lemmatizer.lemmatize_phrase(h) for h in clean_hints]
   Если какая-то подсказка дала [] — WARNING (не error).

Шаг 5. Сборка all_lemmas:
   all_lemmas = term_lemmas + [l for sublist in hints_lemmas for l in sublist]

Шаг 6. Логирование и возврат словаря.

LOGGING
-------
  INFO:    'Предобработка завершена: term_lemmas=..., подсказок=...'
  ERROR:   'Ошибка предобработки: <message>'
  WARNING: при пустых подсказках, дубликатах, пустой лемме

ПРОВЕРКА
--------
  tests/test_preprocess_base.py:
  - term='Ключ!', hints=['Техника', 'Техника'] ->
      term_lemmas=['ключ'], clean_hints=['техника'],
      warnings содержит 'дублирующиеся'
  - Пустой термин ('!!!'): status='error'
  - term='к' * 200: status='error' (слишком длинный)

ПРИМЕЧАНИЕ ПО ТЕСТ-ФАЙЛАМ
-------------------------
В проекте может существовать файл tests/test_preprocess.py из прошлых итераций.
Его следует удалить или очистить: новые тесты размещаются в
test_preprocess_base.py (изм.9) и test_preprocess_full.py (изм.10).
Оставлять пустой test_preprocess.py допустимо, если он не мешает pytest.

ИЗМЕНЕНИЕ 10: Расширенная предобработка с синонимами и весами
--------------------------------------------------------------------------------

ОБЯЗАТЕЛЬНЫЕ ПОЛЯ КОНФИГА (проверить перед реализацией)
-------------------------------------------------------
Изменение 10 использует два поля из Config. Если они отсутствуют в
src/config.py и configs/config.json — добавить до начала реализации:

  use_synonyms: bool          — включить/отключить расширение синонимами.
                                По умолчанию: true
  max_synonyms_per_token: int — максимум синонимов на один исходный токен.
                                По умолчанию: 2

Пример фрагмента configs/config.json:
  "use_synonyms": true,
  "max_synonyms_per_token": 2

ЧТО НУЖНО СДЕЛАТЬ
-----------------
Добавить в src/preprocess.py функцию preprocess_full.
Она вызывает preprocess_base, затем расширяет
результат синонимами и вычисляет веса токенов.

СИГНАТУРА ФУНКЦИИ
------------------
preprocess_full(
  term: str,
  hints: list[str],
  config: Config,
  synonym_dict: SynonymDict,
  lemmatizer: Lemmatizer
) -> dict

ВЫХОДНОЙ СЛОВАРЬ (расширяет базовый)
--------------------------
  {
    ... (все поля из preprocess_base) ...
    'tokens_with_weights': list[tuple[str, float]]
  }

АЛГОРИТМ
--------
Шаг 1. Вызвать preprocess_base(term, hints, config, lemmatizer).
   Если status='error' — сразу вернуть этот error.

Шаг 2. Извлечь: term_lemmas и hints_lemmas.

Шаг 3. Заполнить tokens_weights = [].

Шаг 4. Добавить ТЕРМИН с весом:
   term_w = 0.7 / len(term_lemmas)
   для lemma в term_lemmas: tokens_weights.append((lemma, term_w))

Шаг 5. Добавить ПОДСКАЗКИ с весом:
   total_hint_words = sum(len(lst) for lst in hints_lemmas)
   Если total_hint_words > 0:
     hint_w = 0.3 / total_hint_words
     для sublist в hints_lemmas:
       для lemma в sublist: tokens_weights.append((lemma, hint_w))

Шаг 6. Сбор СИНОНИМОВ С ОБОИХ ИСТОЧНИКОВ (ТЕРМИН + ПОДСКАЗКИ):
  ВАЖНО: синонимы собираются ИЗ ТЕРМИНА И ИЗ ПОДСКАЗОК одновременно.

  Если config.use_synonyms == False:
    пропустить сбор синонимов.
    INFO 'Использование синонимов отключено конфигом'.
    Перейти к шагу 8.

  all_source_lemmas = term_lemmas + [l for sublist in hints_lemmas for l in sublist]
  max_syn = config.max_synonyms_per_token
  unique_synonyms = set()

  для lemma в all_source_lemmas:
    для syn в synonym_dict.get_synonyms(lemma, max_syn):
      unique_synonyms.add(syn)

  Удалить из unique_synonyms синонимы, которые уже есть в tokens_weights
  (чтобы одно слово не получило двойной вес):
  existing_tokens = {t for t, w in tokens_weights}
  unique_synonyms -= existing_tokens

Шаг 7. Добавить синонимы с весом 0.1/M:
   M = len(unique_synonyms)
   Если M > 0:
     syn_w = 0.1 / M
     для syn в unique_synonyms: tokens_weights.append((syn, syn_w))
     INFO 'Добавлено M синонимов'
   Иначе: INFO 'Синонимы не найдены'

Шаг 8. Добавить tokens_with_weights в словарь, вернуть.

ПРИМЕР РАСЧЁТА
--------------
term='ключ', hints=['техника'],
синонимы: 'ключ':['инструмент'], 'техника':['механизм']

  term_lemmas = ['ключ'], term_w = 0.7 -> ('ключ', 0.7)
  hints_lemmas = [['техника']], hint_w = 0.3 -> ('техника', 0.3)
  синонимы из 'ключ': 'инструмент'
  синонимы из 'техника': 'механизм'
  unique_synonyms = {'инструмент', 'механизм'}, M=2, syn_w = 0.05
  tokens_with_weights:
    ('ключ', 0.7)
    ('техника', 0.3)
    ('инструмент', 0.05)
    ('механизм', 0.05)
  Сумма весов: 0.7 + 0.3 + 0.05 + 0.05 = 1.1
  Это нормально: L2-нормирование в векторизаторе (шаг 2) сгладит.
  НЕ исправлять сумму до 1.0 здесь.

ГРАНИЧНЫЕ СЛУЧАИ
----------------
- Терм без подсказок: синонимы собираются из термина, вес 0.3 не прибавляется.
- term состоит из N слов: вес каждого = 0.7/N.
- use_synonyms=False: tokens_with_weights содержит только исходные токены.

ПОДКЛЮЧЕНИЕ К main.py
------------------
В run_pipeline вызвать preprocess вместо заглушки шага 1:

  processed = preprocess(term, hints, config, synonym_dict, lemmatizer)
  если processed['status'] == 'error': вернуть error-ответ
  # Шаг 2 (векторизация) будет добавлен в изм.11

ВАЖНО: ПУБЛИЧНОЕ ИМЯ ФУНКЦИИ
-----------------------------
Вся остальная часть проекта использует имя 'preprocess', а не 'preprocess_full'.
В конце src/preprocess.py добавить строку-псевдоним:

  preprocess = preprocess_full

Это позволяет main.py писать:
  from src.preprocess import preprocess
и получать полную версию с синонимами и весами.
preprocess_base остаётся доступна под своим именем для unit-тестов.

Изменения в run_pipeline делаются постепенно: достаточно добавить
вызов preprocess и убрать заглушку шага 1.
